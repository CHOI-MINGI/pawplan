# PawPlan 알림 설계

작성일: 2026-04-21  
수정일: 2026-04-25  
목적: Flutter 로컬 알림으로 MVP 케어 일정 알림을 안정적으로 제공하기 위한 구조 설계

---

## 1. 왜 별도 설계가 필요한가

`care_schedules`만 있으면 일정은 저장할 수 있다.  
하지만 실제 사용자에게 "내일 예방접종입니다" 같은 알림을 보여주려면 앱 쪽에서 다음 문제가 생긴다.

- 언제 알림을 예약할지 계산해야 한다.
- 일정이 수정되면 기존 로컬 알림을 취소해야 한다.
- 일정이 완료되거나 건너뛰기 처리되면 남은 알림을 취소해야 한다.
- 같은 일정에 중복 알림이 예약되지 않아야 한다.

MVP에서는 서버 푸시가 아니라 `flutter_local_notifications` 기반 기기 내 로컬 알림을 사용한다.  
서버는 일정 데이터를 제공하고, 앱은 해당 일정에 맞춰 로컬 알림을 예약·취소한다.

---

## 2. 설계 목표

- 케어 일정 기반 로컬 알림 예약
- 중복 예약 방지
- 일정 변경 시 로컬 알림 자동 재예약
- 일정 완료·건너뛰기·삭제 시 로컬 알림 자동 취소
- 앱 내 알림 on/off 설정 지원

MVP에서 제외하는 것:

- FCM 기반 서버 푸시
- `user_devices`
- `notification_jobs`
- `notification_logs`
- 크론 기반 발송
- 실패 재시도 큐

---

## 3. 기본 구조

MVP 알림 구조는 2층으로 나눈다.

### 3.1 원본 일정
- `care_schedules`

### 3.2 기기 내 예약 알림
- Flutter 앱 내부에서 `flutter_local_notifications`로 예약
- 알림 식별자는 `schedule_id`와 알림 기준일을 조합해 생성
- 서버 DB에는 로컬 알림 예약 이력을 저장하지 않는다.

---

## 4. 데이터 흐름

```text
care_schedules 조회
-> 앱에서 알림 시점 계산
-> 로컬 알림 예약
-> 일정 수정/완료/건너뛰기/삭제
-> 기존 로컬 알림 취소
-> 필요한 경우 새 로컬 알림 재예약
```

---

## 5. 알림 발송 시점 정책

MVP에서는 너무 많은 규칙을 넣지 말고 단순하게 시작하는 것이 좋다.

### 5.1 기본 알림 정책
- 예정일 1일 전 오전 9시
- 예정일 당일 오전 9시

중요도가 높은 일정만 7일 전 알림을 추가할 수 있지만, 기본 MVP는 2회 알림으로 시작한다.

### 5.2 알림 대상
- `care_schedules.reminder_enabled = true`
- 앱 알림 설정이 켜져 있음
- 기기 알림 권한이 허용됨
- 일정 상태가 `pending`

### 5.3 제외 조건
- 일정 상태가 `completed`
- 일정 상태가 `skipped`
- 일정이 삭제됨
- 보호자가 알림 꺼둠
- 알림 시점이 이미 지남

---

## 6. DB 반영 범위

MVP DB에는 별도 알림 테이블을 만들지 않는다.

필요한 컬럼:

- `care_schedules.reminder_enabled`
- `care_schedules.status`
- `care_schedules.due_date`
- `care_schedules.updated_at`

앱 알림 설정은 MVP에서는 기기 로컬 설정 또는 앱 로컬 저장소로 처리한다.  
서버 동기화가 필요해지는 시점에 사용자 설정 테이블을 추가한다.

---

## 7. 중복 방지 설계

로컬 알림에서 가장 흔한 문제는 중복 예약이다.

### 7.1 로컬 알림 ID 생성 규칙

```text
notificationId = hash("care_reminder:{schedule_id}:{offset_days}")
```

예시:

```text
care_reminder:301:1
care_reminder:301:0
```

앱은 같은 `notificationId`를 취소한 뒤 다시 예약한다.  
이렇게 하면 같은 일정의 같은 시점 알림이 중복 예약되지 않는다.

---

## 8. 앱 동작 방식

MVP에서는 서버 스케줄러 없이 앱에서 알림을 예약한다.

### 8.1 추천 구현 방식
- Flutter + `flutter_local_notifications`
- 일정 목록 조회 후 pending 일정에 대해 로컬 알림 예약
- 일정 상세 수정, 완료, 건너뛰기 시 기존 알림 취소
- 앱 실행 또는 홈 새로고침 시 가까운 일정의 알림 상태를 다시 맞춤

### 8.2 처리 순서
1. `GET /dogs/{dogId}/care-schedules`로 일정 조회
2. `status = pending`이고 `reminder_enabled = true`인 일정 필터링
3. 예정일 1일 전과 당일 오전 9시 알림 시점 계산
4. 기존 `notificationId` 취소
5. 미래 시점인 경우 로컬 알림 예약
6. 일정 완료·건너뛰기·삭제 시 관련 `notificationId` 취소

## 9. 일정 변경 시 처리 규칙

### 9.1 일정 생성
- 기본 알림 정책에 따라 로컬 알림 예약

### 9.2 일정 수정
- 기존 로컬 알림 취소

### 9.3 일정 완료
- 관련 로컬 알림 취소

### 9.4 일정 건너뛰기
- 관련 로컬 알림 취소

### 9.5 일정 삭제
- 관련 로컬 알림 취소

## 10. 예산/행동 유도 알림 확장 가능성

MVP는 케어 일정 알림만 지원해도 충분하지만, 구조상 아래도 확장 가능하다.

- 이번 달 예산 초과 경고
- 장기간 체중 미기록 알림
- 병원 방문 리포트 생성 유도
- 복용약 누락 알림

MVP에서는 모두 로컬 알림으로 처리할 수 있는 범위만 적용한다.  
여러 보호자에게 동시에 보내거나 앱 미실행 상태에서도 서버가 반드시 보내야 하는 알림은 2차 서버 푸시로 넘긴다.

---

## 11. API 설계 반영 포인트

### 11.1 일정 API 응답

일정 응답에는 앱이 로컬 알림을 예약할 수 있도록 아래 값이 포함되어야 한다.

- `id`
- `title`
- `dueDate`
- `status`
- `priority`
- `reminderEnabled`

### 11.2 일정 생성/수정 API의 내부 동작
- `POST /dogs/{dogId}/care-schedules`
- `PATCH /care-schedules/{scheduleId}`
- `POST /care-schedules/{scheduleId}/complete`
- `POST /care-schedules/{scheduleId}/skip`

이 API들은 서버에서 일정 상태만 정확히 변경한다.  
알림 예약·취소는 응답을 받은 Flutter 앱이 로컬에서 처리한다.

### 11.3 2차 확장 API

서버 푸시를 도입할 때만 아래 API를 추가한다.

- `POST /devices`
- `PATCH /devices/{deviceId}`
- `GET /me/notification-settings`
- `PATCH /me/notification-settings`

## 12. 사용자 경험 관점 주의점

- 알림이 너무 많으면 앱을 삭제한다.
- 기본 MVP는 1일 전과 당일 2회 알림으로 제한한다.
- 앱 설정에서 케어 알림 on/off를 바꿀 수 있어야 한다.
- 일정 완료 후 같은 일정 알림이 다시 오는 일은 반드시 막아야 한다.

---

## 13. MVP 구현 권장 범위

### 13.1 반드시 구현
- `flutter_local_notifications` 설정
- Android 알림 권한 요청
- 일정 조회 후 로컬 알림 예약
- 일정 완료/수정/건너뛰기 시 로컬 알림 취소 또는 재예약
- 같은 일정 중복 알림 방지
- 앱 내 케어 알림 on/off 설정

### 13.2 있으면 좋음
- 중요 일정 7일 전 알림
- 알림 클릭 시 일정 상세 화면 이동
- 홈 진입 시 가까운 일정 알림 재동기화

### 13.3 후순위
- `user_devices`
- `notification_jobs`
- `notification_logs`
- FCM 연동
- 크론 기반 발송
- 실패 재시도
- 예산 경고 알림
- 복용약 세부 알림
- 이메일 대체 발송

## 14. 검증 방법

### 14.1 기능 검증
- 일정 생성 후 예정일 1일 전/당일 알림이 예약되는지 확인
- 일정 수정 시 기존 로컬 알림이 취소되고 새 알림이 예약되는지 확인
- 일정 완료 시 남은 로컬 알림이 취소되는지 확인
- 같은 일정의 같은 시점 알림이 중복 예약되지 않는지 확인
- 알림 권한이 꺼져 있을 때 앱이 적절히 안내하는지 확인

### 14.2 사용자 검증
- 알림 빈도가 과하지 않은지
- 실제로 필요한 순간에 오는지
- 일정 완료 후 불필요한 알림이 다시 오지 않는지

---

## 15. 한 줄 정리

PawPlan의 알림 시스템은  
MVP에서는 `care_schedules`를 기준으로 Flutter 앱이 로컬 알림을 예약·취소하는 구조가 가장 현실적이다.  
서버 푸시, FCM, 알림 작업 테이블은 다중 보호자와 실서비스 운영이 필요해질 때 2차로 확장한다.
