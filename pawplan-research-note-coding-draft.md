# PawPlan 이번 주차 연구노트

작성일: 2026년 5월 10일  
대상 기간: 2026년 5월 8일 ~ 2026년 5월 14일  
프로젝트명: PawPlan  
작성 기준: 로그인 및 기본 서버 연동 단계  

---

## 1. 최민기 - 백엔드 개발(Express 중심)

### 1. 목적

이번 주차 목표는 PawPlan의 백엔드 서버 기본 구조를 구성하고, Flutter 앱에서 사용할 수 있는 인증 API 흐름을 준비하는 것이다. 특히 회원가입, 로그인, JWT 인증 구조를 먼저 잡아 이후 반려견 등록과 기록 관리 API가 사용자별로 분리될 수 있도록 하는 데 목적을 두었다.

### 2. 연구방법

Express와 TypeScript를 기반으로 REST API 서버 구조를 검토하였다. 인증 기능은 이메일과 비밀번호를 입력받고, 로그인 성공 시 JWT access token을 발급하는 방식으로 설계하였다. 비밀번호는 평문 저장을 피하고 해시 처리하는 방향으로 정리하였다.

### 3. 도출결과

백엔드 구현 순서는 `서버 기본 구조 -> 인증 API -> 인증 미들웨어 -> 반려견 API` 순서가 적절하다고 판단하였다. 또한 앱과 연동하기 쉽도록 성공/실패 응답 형식을 통일해야 한다는 결론을 얻었다.

### 4. 문제점 분석

프론트엔드에서 사용하는 요청 필드와 백엔드에서 검증하는 필드가 다르면 로그인 연동 과정에서 오류가 발생할 수 있다. 또한 Android 에뮬레이터에서 서버에 접근할 때 `localhost`가 아닌 `10.0.2.2`를 사용해야 하므로 서버 바인딩 설정도 함께 확인해야 한다.

### 5. 개선방안 및 향후 계획

다음 작업에서는 인증 API를 우선 안정화하고, 반려견 등록 및 조회 API를 연결할 계획이다. API 명세 변경이 생길 경우 정재운과 공유하여 DB/API 문서를 함께 갱신한다.

## 2. 박상민 - 프론트엔드 개발(Flutter 중심)

### 1. 목적

이번 주차 목표는 Flutter 앱에서 로그인 화면과 기본 API 호출 구조를 준비하는 것이다. 사용자가 이메일과 비밀번호를 입력하여 서버에 로그인 요청을 보내고, 이후 화면으로 이동할 수 있는 흐름을 만드는 것을 목표로 하였다.

### 2. 연구방법

Flutter의 Material UI를 기준으로 로그인 화면 구성을 검토하였다. 서버 통신 코드는 화면 코드와 분리하여 API client 형태로 관리하는 방법을 사용하기로 하였다. Android 에뮬레이터 실행 시 API 주소는 `http://10.0.2.2:4000/api/v1`을 사용하는 것으로 정리하였다.

### 3. 도출결과

앱의 초기 흐름은 `로그인 화면 -> 인증 요청 -> 토큰 저장 -> 홈 화면 이동`으로 구성하는 것이 적절하다고 판단하였다. 또한 로그인 실패, 서버 연결 실패, 입력값 누락 상황에 대한 안내 처리가 필요하다는 점을 확인하였다.

### 4. 문제점 분석

에뮬레이터와 실제 기기에서 서버 주소가 달라질 수 있어 테스트 환경마다 API base URL을 바꾸어야 하는 문제가 있다. 또한 백엔드 응답 형식이 바뀌면 Flutter 모델과 화면 처리 로직도 함께 수정해야 한다.

### 5. 개선방안 및 향후 계획

API 주소를 빌드 옵션이나 설정값으로 분리하고, 로그인 화면에 로딩/오류 상태를 추가할 계획이다. 다음 주차에는 반려견 온보딩 화면과 홈 화면을 이어서 구현한다.

---

## 3. 최지훈 - 프로젝트 총괄/기획/E2E 테스트

### 1. 목적

이번 주차 목표는 팀별 역할과 개발 순서를 다시 확인하고, 로그인부터 기본 데이터 조회까지의 E2E 테스트 흐름을 정리하는 것이다. 또한 연구노트, 세부공정표, 발표용 설명 자료를 정리하여 이후 개발 과정에서 기준으로 사용할 수 있도록 하는 데 목적을 두었다.

### 2. 연구방법

프로젝트 계획서, 세부공정표, API 명세서, DB 설계서를 기준으로 현재 주차에서 확인해야 할 흐름을 정리하였다. 테스트 관점에서는 사용자가 앱을 실행한 뒤 로그인하고, 서버와 정상적으로 통신하는지 확인하는 최소 흐름을 우선 대상으로 잡았다.

### 3. 도출결과

이번 주차의 핵심 점검 흐름은 `앱 실행 -> 로그인 요청 -> 토큰 발급 -> 인증 API 호출 가능 여부 확인`으로 정리하였다. 또한 각 담당자의 역할을 최민기 백엔드, 박상민 프론트엔드, 정재운 DB/API 설계, 최지훈 총괄 및 E2E 테스트로 명확히 구분하였다.

### 4. 문제점 분석

문서상 일정과 실제 개발 속도가 달라질 수 있으므로, 연구노트와 공정표의 표현을 너무 완료 중심으로 작성하면 이후 주차 내용과 충돌할 수 있다. 또한 E2E 테스트는 백엔드와 프론트엔드가 모두 준비되어야 하므로, 한쪽의 지연이 전체 검증 일정에 영향을 줄 수 있다.

### 5. 개선방안 및 향후 계획

각 기능이 완료될 때마다 E2E 테스트 체크리스트를 갱신하고, API 응답 형식이나 DB 필드 변경 사항을 문서에 반영할 계획이다. 다음 주차에는 반려견 등록과 일정 조회 흐름을 중심으로 테스트 범위를 확장한다.

---

## 4. 정재운 - 백엔드 DB/API 설계(DB 중심)

### 1. 목적

이번 주차 목표는 PawPlan의 핵심 데이터 구조를 실제 API 구현과 연결할 수 있도록 DB schema와 API 필드 기준을 정리하는 것이다. 특히 사용자, 반려견, 일정, 기록 데이터가 서로 어떤 관계로 연결되는지 명확히 하는 것을 목표로 하였다.

### 2. 연구방법

PostgreSQL과 Prisma를 기준으로 데이터 모델을 검토하였다. MVP 단계에서 필요한 핵심 테이블을 먼저 정리하고, Prisma migration을 통해 팀원들이 동일한 DB 구조를 사용할 수 있도록 하는 방식을 선택하였다.

### 3. 도출결과

초기 구현에 필요한 핵심 데이터는 사용자 계정, 반려견 프로필, 케어 일정, 건강 기록, 병원 방문 기록, 지출 기록으로 정리하였다. API 구현 시에는 각 데이터가 `user_id` 또는 `dog_id`를 기준으로 연결되어야 한다는 점을 확인하였다.

### 4. 문제점 분석

DB 설계에서 필드명을 자주 바꾸면 백엔드 API와 Flutter 모델까지 함께 수정해야 하는 문제가 있다. 또한 날짜와 시간 데이터는 일정, 건강 기록, 병원 방문 기록에서 모두 사용되기 때문에 형식을 통일하지 않으면 조회 화면에서 정렬 오류가 발생할 수 있다.

### 5. 개선방안 및 향후 계획

Prisma schema를 기준 문서처럼 관리하고, API 명세와 필드명이 일치하는지 계속 확인할 계획이다. 다음 작업에서는 반려견 등록과 케어 일정 생성에 필요한 테이블 관계를 우선 검증한다.

# PawPlan 어플리케이션 소개 대본

## 1. 발표용 대본

안녕하세요. 컴퓨터공학과 종합프로젝트 6조 팀장 최지훈입니다.
저희 팀은 반려견 보호자를 위한 건강관리 및 비용관리 앱, PawPlan을 개발하고 있습니다.

반려견을 키우는 보호자는 예방접종, 심장사상충 예방, 구충, 건강검진, 복약, 병원 방문, 사료와 용품 구매처럼 반복적으로 챙겨야 할 일이 많습니다. 하지만 실제로는 이런 정보가 캘린더, 메모장, 사진첩, 병원 영수증, 가계부 등에 흩어져 관리되는 경우가 많습니다. 그래서 중요한 예방 일정을 놓치거나, 병원에 방문했을 때 최근 증상과 이전 진료 내용을 바로 설명하기 어려운 문제가 생깁니다. 또한 한 달에 반려견에게 어느 정도 비용이 들어가는지 파악하기도 쉽지 않습니다.

PawPlan은 이런 문제를 해결하기 위해 반려견을 중심으로 건강 기록, 병원 방문 기록, 지출 내역, 케어 일정을 한 곳에서 관리할 수 있도록 만든 모바일 앱입니다. 사용자는 앱에 로그인한 뒤 반려견을 등록하고, 홈 화면에서 이번 달 지출, 예정 일정, 예상 비용을 확인할 수 있습니다. 기록 화면에서는 건강 로그, 병원 방문 기록, 지출 내역을 따로 입력하면서도, 전체 흐름은 통합 타임라인으로 확인할 수 있도록 구성했습니다.

저희 앱의 핵심 기능은 크게 네 가지입니다. 
첫 번째는 예방 일정 관리입니다. 예방접종, 구충, 검진 같은 일정을 등록하고 완료 또는 건너뛰기 상태를 관리할 수 있습니다. 
두 번째는 건강 및 병원 기록 관리입니다. 체중, 증상, 복약, 진료 내용 등을 기록하여 반려견의 상태 변화를 추적할 수 있습니다. 
세 번째는 지출 관리와 비용 예측입니다. 병원비나 사료비 같은 지출을 입력하면 월별 지출과 향후 예상 비용을 확인할 수 있습니다. 
네 번째는 병원 방문 리포트입니다. 보호자가 입력한 최근 기록을 바탕으로 병원 방문 전에 참고할 수 있는 요약 정보를 제공합니다.

기술적으로는 Flutter를 사용하여 Android 모바일 앱을 구현하고, 백엔드는 Express와 TypeScript 기반 REST API로 구성합니다. 
데이터베이스는 PostgreSQL을 사용하며 Prisma를 통해 schema와 migration을 관리합니다. 
알림 기능은 서버 푸시가 아니라 Flutter 로컬 알림을 사용하여 MVP 범위 안에서 구현 난이도를 조절했습니다.
현재는 다음에서 보실 수 있듯이 프론트는 Android Emulator에 Google의 Pixel 10에서 실행되고 있으며 백엔드는 Docker 환경에서 실행되고 있습니다.

저희 프로젝트의 목표는 단순히 기록을 저장하는 앱을 만드는 것이 아니라, 보호자가 반려견의 건강관리와 비용 계획을 더 쉽게 이어갈 수 있도록 돕는 실용적인 관리 도구를 만드는 것입니다.

## 2. 시연 흐름 대본

먼저 앱을 실행하면 PawPlan 로그인 화면이 나타납니다. 회원가입 기능도 동작하나, 오늘은 목업 데이터가 작성되어 있는 데모 계정으로 보여드리겠습니다. 
데모 계정으로 로그인하면 등록된 반려견인 초코의 홈 화면으로 이동합니다.

우선 애완동물 데이터부터 살펴보겠습니다.
현재는 초코라는 목업 데이터로 정보가 입력되어 있습니다.
이름부터 견종, 생년월일, 성별, 중성화 여부, 체중과 기타 사항들에 대해서 정보 입력 및 수정이 가능합니다.

다음은 가족 공유 기능입니다.
여러 명의 보호자가 같은 반려동물에 대해 관리 및 데이터를 입력할 경우를 대비하여 가족 공유 기능을 추가했습니다.
현재는 가입 이메일 기반으로 가족 공유 기능에 초대할 수 있으며, 역할은 viewer, editor, owner 3종으로 분류를 해두었습니다.
각 역할마다 엑세스 권한이 다릅니다.
viewer는 데이터에 대한 조회는 가능하나, 데이터 작성 및 수정 권한은 없습니다.
editor는 데이터에 대한 조회와 추가 및 수정은 가능하나, 반려동물 삭제 등 주요 데이터 삭제 기능은 없습니다.
owner는 데이터에 대한 추가, 수정, 삭제 모든 권한을 가지고 있습니다.

다음은 데이터 삭제 기능입니다.
실수로 삭제되는 것을 방지하기 위해서 기존에 등록한 반려동물의 이름을 작성하지 않으면 삭제가 되지 않도록 하였습니다.
이 부분은 반려동물 사망 등의 이류를 기반으로 추모 탭을 추가로 제작하여 해당 탭으로 데이터를 옮겨갈 수 있는 기능을 구상 중에 있습니다.
사랑했던 반려동물의 데이터가 남아있어, 어플의 장기 이용자가 늘어날 수 있지 않을까, 또한 새로운 반려동물을 키우게 된다면 재진입 유저 수를 늘릴 수 있지 않을까 하여 생각한 기능 중 하나입니다.
리포트 버튼은 추후 리포트 탭 설명 드리며 같이 보여드리겠습니다.

이제 메인 홈 화면입니다.
홈 화면에서는 반려견의 기본 정보와 이번 달 지출, 예정 일정, 6개월 예상 비용을 한눈에 확인할 수 있습니다. 이 화면은 보호자가 앱을 열었을 때 지금 바로 확인해야 할 정보를 빠르게 보여주는 역할을 합니다.
현재 UI 부분이 조금 깨지는 것이 확인 되는데, 다양한 기종에서 사용 가능하도록 반응형 구조를 적용하다보니 이번 주에 나타는 오류 중 하나입니다.
프론트엔드 UI 개발 담당인 박상민 팀원이 이슈 트래킹 중입니다.

다음으로 기록 탭에서는 건강 기록, 병원 방문 기록, 지출 기록을 확인할 수 있습니다. 각각의 기록은 따로 저장되지만, 사용자는 통합 타임라인을 통해 날짜순으로 전체 흐름을 볼 수 있습니다. 
이를 통해 보호자는 최근에 어떤 증상이 있었는지, 병원 방문 후 어떤 지출이 발생했는지 연결해서 파악할 수 있습니다.
또한 운영진은 회원가입 시에 입력되는 정보인 견종에 따른 데이터를 통해 어떠한 견종에서 어떠한 건강 이슈가 다발적으로 발생했는지 데이터 수집이 가능해집니다.
이를 통해 저희 pawplan의 메인 기능 중 하나인 향후 예상 비용 예측 시스템을 더욱 고도화할 수 있을 것으로 기대하고 있습니다.

다음은 정보 탭입니다.
정보 탭에서는 반려견의 건강 상태와 복약 정보를 관리할 수 있습니다. 
예를 들어 알레르기, 기저질환, 복용 중인 약을 정리해두면 병원 방문 시 참고 자료로 활용할 수 있습니다.

또한 MVP 2차에 기록되어 있는 목적 중 하나입니다만, 각 동물병원으로도 건강 정보를 export 할 수 있도록 할 예정입니다.
그 이유 중 대표적인 것으로, 애완동물들은 사람과 달리 의료 기록이 별도로 기록되지 않습니다.
내원하는 병원이 바뀐다면, 보호자의 기억 또는 따로 보관한 처방전에 의존하여 이전에 방문했던 병원에서 받은 처방과 의료기록, 건강상태를 알려야합니다.
이러한 사항들을 새로이 방문할 병원 예약 시에 쉽게 전달할 수 있다면 보호자들의 불편함 해소와 더불어 수의사들은 정확한 진료에 도움이 될 기반이 있습니다.

또한 주요 수익 모델 중 하나로 동물 보험과의 연계도 기대하고 있습니다.
현재 동물 보험은 보호자 입장에서 몇 가지 불편한 점이 있습니다. 먼저 보험 상품마다 보장 범위, 자기부담금, 보장 제외 항목, 갱신 조건이 달라서 어떤 보험이 우리 반려견에게 적합한지 비교하기 어렵습니다. 예를 들어 반려견의 나이, 견종, 기존 질환 여부, 병원 방문 빈도에 따라 필요한 보장 내용이 달라질 수 있는데, 보호자가 이를 직접 판단하기에는 정보가 분산되어 있습니다.

두 번째 문제는 보험금 청구 과정입니다. 보험금을 청구하려면 진료비 영수증, 진료 내역, 처방 정보, 검사 결과 같은 자료를 따로 모아야 하는데, 대부분의 보호자는 이러한 자료를 사진첩이나 종이 영수증 형태로 보관합니다. 이 과정에서 자료가 누락되거나, 언제 어떤 증상으로 병원에 방문했는지 기억에 의존해야 하는 상황이 발생합니다. 결과적으로 보험을 가입했더라도 청구 과정이 번거로워 실제 활용도가 떨어질 수 있습니다.

보험사 입장에서도 한계가 있습니다. 반려견의 예방접종 이력, 만성질환 여부, 병원 방문 패턴, 월별 지출 데이터가 표준화되어 있지 않으면 사용자에게 적합한 보험 상품을 추천하거나, 합리적인 리스크 평가를 하기 어렵습니다. 데이터가 부족하면 보험사는 보수적으로 상품을 설계할 수밖에 없고, 보호자는 자신의 반려견에게 맞는 상품을 찾기 어려워지는 문제가 생깁니다.

PawPlan은 이 문제를 해결할 수 있는 기반을 제공합니다. 앱 안에서 건강 기록, 병원 방문 기록, 지출 내역, 예방 일정, 첨부파일을 반려견 단위로 구조화해서 관리하기 때문에, 향후 사용자의 명시적 동의를 전제로 보험 청구에 필요한 자료를 정리하거나, 반려견 상태에 맞는 보험 상품을 추천하는 기능으로 확장할 수 있습니다. 예를 들어 병원 방문 기록과 영수증 첨부 내역을 바탕으로 청구 서류 목록을 자동으로 정리하거나, 나이와 건강 기록, 최근 지출 패턴을 기준으로 보험 필요성을 안내할 수 있습니다.

수익 모델 측면에서는 보험 비교 및 추천을 통한 제휴 수수료, 보험 상담 연결 수수료, 보험사와의 B2B 데이터 연동 모델을 기대할 수 있습니다. 단, 이때 건강 기록과 지출 데이터는 민감한 정보이기 때문에 반드시 사용자 동의, 접근 권한 관리, 비식별화 또는 최소 정보 제공 원칙을 전제로 해야 합니다. 즉 PawPlan은 단순히 보험 광고를 붙이는 방식이 아니라, 보호자가 이미 입력한 기록을 바탕으로 보험 선택과 청구 과정을 쉽게 만들어주는 방향으로 수익 모델을 정립할 수 있을 것으로 기대하고 있습니다.

마지막으로 리포트 화면에서는 병원 방문 리포트와 비용 예측 정보를 확인할 수 있습니다.
병원 방문 리포트는 보호자가 입력한 최근 건강 기록과 병원 기록을 바탕으로 요약 정보를 제공하며, 비용 예측 기능은 앞으로 발생할 수 있는 지출 규모를 대략적으로 파악하는 데 도움을 줍니다.
해당 기능은 AI 기반이 아닌 단순 룰 베이스 모델입니다.
룰 베이스 모델이란 학습 데이터로 결과를 예측하는 방식이 아니라, 미리 정한 조건과 계산 규칙에 따라 결과를 산출하는 방식입니다. PawPlan에서는 반려견의 나이, 체중, 중성화 여부, 활동량, 예방 일정, 최근 병원 방문 기록, 최근 지출 내역, 지출 카테고리 같은 정보를 기준으로 예상 비용을 계산하는 구조를 사용합니다.

예를 들어 예방접종이나 심장사상충 예방처럼 주기적으로 발생하는 항목은 일정 주기를 기준으로 비용을 반영할 수 있습니다. 병원비나 약값처럼 변동성이 큰 항목은 최근 입력된 지출 기록을 기준으로 월별 평균을 계산하고, 건강 상태나 병원 방문 빈도에 따라 일정한 보정값을 적용할 수 있습니다. 이렇게 계산된 결과는 월 예상 비용, 6개월 예상 비용, 연간 예상 비용처럼 보호자가 이해하기 쉬운 형태로 보여줄 수 있습니다.

저희가 처음부터 AI 모델을 사용하지 않은 이유는 세 가지입니다. 첫째, MVP 단계에서는 충분한 학습 데이터가 확보되어 있지 않습니다. AI 모델을 사용하려면 많은 사용자 데이터와 실제 비용 데이터가 필요하지만, 초기 프로젝트 단계에서는 데이터가 부족하기 때문에 AI 예측 결과의 신뢰도를 보장하기 어렵습니다.

둘째, 반려동물 건강과 비용 예측은 설명 가능성이 중요합니다. 보호자는 단순히 "앞으로 얼마가 나올 것 같습니다"라는 결과만 보는 것이 아니라, 왜 그런 예상이 나왔는지 이해할 수 있어야 합니다. 룰 베이스 방식은 예방 일정, 최근 지출, 병원 방문 횟수 등 어떤 요소가 비용에 반영되었는지 설명하기 쉽습니다.

셋째, 의료적 판단으로 오해될 위험을 줄일 수 있습니다. AI 기반으로 건강 위험이나 병원비를 예측한다고 표현하면 사용자가 이를 진단이나 의학적 판단처럼 받아들일 수 있습니다. 하지만 PawPlan의 비용 예측은 보호자가 입력한 기록을 바탕으로 비용 계획을 보조하는 기능이며, 진단이나 치료 결정을 대신하지 않습니다. 그래서 초기 버전에서는 명확한 규칙 기반 계산 방식을 선택했습니다.

또한 룰 베이스 모델은 개발과 검증이 비교적 쉽다는 장점이 있습니다. 계산식이 명확하기 때문에 테스트 케이스를 만들기 쉽고, 특정 항목의 비용이 과도하게 반영되면 규칙을 수정하여 빠르게 개선할 수 있습니다. 이후 실제 사용자 데이터가 충분히 쌓이면 현재의 룰 베이스 결과를 기준선으로 삼아 통계 기반 모델이나 AI 기반 예측 모델로 고도화할 수 있습니다. 즉 현재 구조는 AI를 배제하기 위한 선택이 아니라, 초기 MVP에서 안정성과 설명 가능성을 확보한 뒤 점진적으로 고도화하기 위한 출발점입니다.

이처럼 PawPlan은 예방 일정, 건강 기록, 병원 기록, 지출 관리를 하나의 앱 안에서 연결하여 반려견 보호자의 반복적인 관리 부담을 줄이는 것을 목표로 합니다.

## 3. 짧은 소개 문구

PawPlan은 반려견의 예방 일정, 건강 기록, 병원 방문 기록, 지출 내역을 한 곳에서 관리하는 모바일 케어 앱입니다. 보호자는 반려견별 기록을 통합 타임라인으로 확인하고, 병원 방문 전 리포트와 예상 비용 정보를 통해 더 체계적으로 반려견을 관리할 수 있습니다.

이상 6조 발표 마치겠습니다. 감사합니다.
